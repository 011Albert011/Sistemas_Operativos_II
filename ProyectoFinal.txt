\section{Introducción.}

El presente documento detalla el funcionamiento y la lógica de programación detrás de AutoHub, una plataforma web diseñada para demostrar la aplicación práctica de conceptos fundamentales de Sistemas Operativos. El proyecto utiliza un stack tecnológico integrado por PHP para la lógica de negocio, así como el control de dispositivos de entra y salida, así como la administración del Sistema de Archivos \textbf{(File System)} mediante la creación de tickets de compra en formato .PDF, que a comparación de la practica anterior, se opta esta actualización para una mejor manipulación del archivo creado, por ultimo, la gestión de comunicaciones para el envío de información vía correo electrónico.\\

La estructura y el diseño se fundamentan en \textbf{HTML5} y \textbf{CSS3}, mientras que el dinamismo se logra mediante una integración entre \textbf{PHP} y \textbf{JavaScript}. Esta conexión permite la manipulación de datos en tiempo real para generar archivos tipo PDF que registran la persistencia de las transacciones. Posteriormente, se gestiona la recuperación de estos archivos para su envío al usuario en sesión, garantizando la trazabilidad de la información, integrando un botón para la impresión del ticket y que el usuario pueda tener un control completo de su información de copra.\\

Esta práctica tiene como propósito implementar la teoría del Control de Dispositivos vista en clase, resolviendo problemas de almacenamiento, organización y recuperación de datos. Se pone especial énfasis en el control de dispositivos, siendo mas específicos, una impresora, facilitando la manipulación completa de la información de compra del usuario.

\section{Arquitectura del Sistema.}
En esta sección se detalla cómo AutoHub gestiona el control de dispositivos, creando un puente por medio de PHP desde la entrada de datos por mouse, procesamiento en base de datos, gestión de archivos en disco, hasta la salida física por hardware.

La arquitectura se basa en una jerarquía de capas de abstracción. El flujo inicia en la Capa de Aplicación \textbf{(JavaScript)}, escala a la Capa de Servicios \textbf{(PHP/Servidor Web)} y finalmente interactúa con el Kernel del Sistema Operativo para gestionar el acceso a periféricos de salida \textbf{(impresora)} mediante llamadas al sistema.

El proyecto implementa una arquitectura basada en Middleware, facilitando la interoperabilidad entre el frontend (JS) y el backend (PHP/SQL). Se utiliza JSON como mecanismo de mensajería para el transporte de datos, y se han integrado capas de seguridad mediante la gestión de sesiones y un monitor de validación de stock, asegurando que la disponibilidad de recursos sea consistente en todo el ecosistema de la aplicación.

\subsection{Modelo de capas / MVC.}
AutoHub sigue una arquitectura en capas similar a un patrón MVC básico:
\begin{itemize}
    \item \textbf{Vista / Presentación:} HTML, CSS y JavaScript en el navegador que muestran los productos, el carrito y el formulario de compra.
    \item \textbf{Controlador / Lógica de negocio:} PHP recibe solicitudes HTTP, procesa la compra, valida la sesión, genera tickets y envía correos.
    \item \textbf{Modelo / Datos:} MySQL almacena usuarios, inventario, transacciones y registros de compra.
\end{itemize}

\begin{center}
\begin{tabular}{c}
Cliente (HTML/CSS/JS) \\
\(\downarrow\) \\
Servidor Apache + PHP \\
\(\downarrow\) \\
Base de datos MySQL \\
\(\downarrow\) \\
Sistema de archivos / impresora
\end{tabular}
\end{center}

Este enfoque separa responsabilidades, facilita la evolución del sistema y mejora la seguridad porque el cliente no toma decisiones críticas sobre la base de datos ni el almacenamiento.

\subsection{Seguridad desde el diseño.}
La seguridad se integra desde la arquitectura:
\begin{itemize}
    \item \textbf{Principio de mínimo privilegio:} El servicio PHP debe utilizar un usuario de base de datos con permisos restringidos únicamente a las operaciones necesarias (SELECT, INSERT, UPDATE, DELETE) sobre las tablas de la aplicación.
    \item \textbf{Aislamiento de recursos:} El backend solo escribe en la carpeta \texttt{tickets/} y no en otras áreas del servidor, reduciendo el riesgo de modificación no autorizada.
    \item \textbf{Seguridad de sesiones:} Las variables de sesión se usan para autenticar al usuario antes de permitir acciones sensibles como comprar, generar ticket o imprimir.
    \item \textbf{Validación server-side:} Todas las entradas se revisan en PHP, no solo en JavaScript, para evitar inyección de datos y manipulación del cliente.
\end{itemize}

\subsubsection{Análisis de Seguridad del Sistema.}
El sistema AutoHub implementa múltiples capas de seguridad para proteger la integridad de los datos, la autenticación de usuarios y la prevención de ataques comunes. A continuación, se presenta un reporte detallado del estado de seguridad actual, incluyendo vulnerabilidades identificadas, medidas implementadas y controles preventivos.

\begin{itemize}
    \item \textbf{Prevención de Inyección SQL (SQLi):} Se identificaron consultas SQL vulnerables a inyección en módulos críticos como login, registro y procesamiento de pagos. Para mitigar este riesgo, se implementaron \textit{prepared statements} utilizando \texttt{mysqli\_prepare()}, \texttt{mysqli\_stmt\_bind\_param()} y \texttt{mysqli\_stmt\_execute()}. Esta técnica separa la lógica SQL de los datos de entrada, eliminando la posibilidad de ejecución de código malicioso a través de parámetros concatenados.
    
    \item \textbf{Autenticación Segura con Hashing de Contraseñas:} Las contraseñas se almacenaban inicialmente en texto plano, exponiendo credenciales en caso de brecha de base de datos. Se implementó hashing fuerte usando \texttt{password\_hash()} con algoritmo \texttt{PASSWORD\_BCRYPT}, que genera hashes irreversibles con sal automática. La verificación se realiza mediante \texttt{password\_verify()}, asegurando compatibilidad con hashes existentes y migración automática de contraseñas legadas al primer login exitoso.
    
    \item \textbf{Protección contra Ataques de Sesión:} Se detectó vulnerabilidad a \textit{session fixation}, donde un atacante podría fijar un ID de sesión conocido. Tras autenticación exitosa, se regenera el ID de sesión con \texttt{session\_regenerate\_id(true)}, invalidando sesiones potencialmente comprometidas y fortaleciendo la protección contra robo de sesiones.
    
    \item \textbf{Integridad Transaccional en Operaciones Críticas:} En el módulo de pagos, operaciones múltiples (crear ticket, actualizar stock, insertar detalles) se ejecutaban sin atomicidad, risking inconsistencias en caso de error. Se implementaron transacciones atómicas usando \texttt{mysqli\_begin\_transaction()}, \texttt{mysqli\_commit()} y \texttt{mysqli\_rollback()}, garantizando que todas las operaciones se completen exitosamente o se reviertan por completo, manteniendo la consistencia de la base de datos.
    
    \item \textbf{Validación y Sanitización de Entradas:} Entradas de usuario no se validaban adecuadamente, permitiendo datos malformados o maliciosos. Se agregó sanitización de correos electrónicos con \texttt{filter\_var()} y conversión explícita de tipos (int, float) antes de consultas SQL, previniendo inyecciones y errores de tipo que podrían comprometer la lógica de negocio.
    
    \item \textbf{Auditoría y Monitoreo de Eventos:} Para mejorar la trazabilidad, se preparó un sistema básico de logs para registrar eventos críticos como intentos de login fallidos, pagos procesados y actualizaciones de stock. Esto facilita la detección de anomalías y el cumplimiento de requisitos de auditoría, aunque su implementación completa queda pendiente para futuras iteraciones.
\end{itemize}

Estas implementaciones fortalecen la autenticación fuerte, permisos granulares y prevención de vulnerabilidades comunes como SQLi, XSS (mediante \texttt{htmlspecialchars()}) y CSRF (preparado para tokens).


\subsection{Control de acceso y roles.}
Para mantener la integridad del sistema se definen roles y niveles de acceso:
\begin{itemize}
    \item \textbf{Usuario autenticado:} Puede navegar, gestionar su carrito, pagar y recibir el comprobante.
    \item \textbf{Administrador / Gestión:} Si existe, debe tener acceso solo a funciones de inventario y consulta de registros, no al manejo de datos sensibles que no le correspondan.
\end{itemize}

Cada petición sensible en el servidor debe verificar el rol y el estado de la sesión antes de permitir cualquier modificación de datos.

\subsection{Conexion entre lenguajes (PHP y JavaScript).}
El sistema implementa una comunicación cliente-servidor utilizando \textbf{JSON} como formato de intercambio de datos. Esta comunicación permite transferir la información del carrito de compras, gestionada como un array en el frontend \textbf{(JavaScript)}, hacia el backend \textbf{(PHP)} para su procesamiento. Una vez recibidos, los datos son insertados en la base de datos, lo que posibilita la generación del comprobante \textbf{(ticket)} y su posterior envío por correo electrónico, a su vez redirigiendo al usuario para que pueda hacer la impresión del \textbf{ticket}.

El uso de la API fetch permite una comunicación asíncrona. Esto evita el bloqueo del hilo principal de ejecución del navegador mientras el backend procesa la transacción, garantizando que la interfaz de usuario se mantenga responsiva mientras el SO gestiona las operaciones de I/O en segundo plano.

\subsection{Persistencia de Datos.}
La principal distinción entre el proyecto previo y la implementación actual radica en el enfoque adoptado para la persistencia de los datos. Mientras que en el primer proyecto la información era gestionada de forma volátil, almacenándose únicamente en memoria temporal durante la sesión activa en el servidor, la nueva versión introduce un mecanismo de almacenamiento permanente.

En esta implementación, se ha optado por generar y organizar archivos de texto que registran de manera estructurada los detalles de cada compra realizada por el cliente, incluyendo los productos adquiridos y el monto total correspondiente. Este enfoque no solo garantiza la conservación de la información más allá de la sesión actual, sino que también sienta las bases para futuras funcionalidades como el historial de compras o la generación de reportes.

\subsection{Permisos de Acceso.}
Para garantizar que el sistema pueda crear y modificar los archivos necesarios dentro del directorio \texttt{tickets}, es fundamental configurar adecuadamente los permisos de escritura en el sistema de archivos.

En entornos Windows que utilizan WAMP (Windows, Apache, MySQL, PHP), los permisos se asignan a través de las propiedades del directorio. Específicamente, se debe conceder al usuario bajo el cual se ejecuta el servicio de Apache los permisos de \textit{Modificar} y \textit{Escribir} sobre la carpeta destinada al almacenamiento de los tickets. Esta configuración asegura que el backend tenga los privilegios necesarios para la creación y modificación de archivos durante la ejecución del sistema.

\subsection{Administración de Entrada y Salida.}
Para la salida física, el sistema utiliza un puente de software (Imprimir.php) que actúa como interfaz entre el navegador y el hardware. Al ejecutar la función window.print(), se invoca al Spooler de Impresión del Sistema Operativo. Este componente gestiona una cola de trabajos, permitiendo que el CPU se libere rápidamente mientras el Driver de la impresora traduce el archivo PDF a lenguaje de control de hardware (ESC/P) para su impresión final.

\subsection{Monitor de Integridad y Validación de Recursos (Middleware).}
Cuando se realiza la compra, ya no solo se ejecuta y se manda el ticket, ahora al implementar un inventario, lo que hacemos es interceptar la petición y realizamos una verificación previa. Se revisa (vía SQL) cuantas unidades quedan realmente de ese vehículo, esta consulta es vital, ya que el carrito de compras usa memoria volátil, siendo nuestra base de datos la única fuente de datos reales.

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
\newpage
\thispagestyle{ARC}

\begin{itemize}
    \item \textbf{JavaScript:} Por parte del cliente tenemos la gestión de la interacción con el usuario en tiempo real.

    \item \textbf{PHP:} El servidor, que procesa la lógica y las reglas del sistema, actuando el \textbf{Middleware de integración}, permitiendo que una acción en la interfaz (JS), resulte en un cambio físico en el disco duro (SQL).

    \item \textbf{SQL:} Los datos que mantienen la persistencia y el estado del inventario.

    \item \textbf{Servicio de Mensajería:} JSON es un protocolo de intercambio. Sin él, PHP y JavaScript no podrían compartir estructuras de datos complejas (como una lista de varios autos con diferentes cantidades). El flujo sería: \textit{JS (JSON.stringify) -> HTTP POST -> PHP (jsondecode)}.

    Al usarlo de esta manera se garantiza que la información es independiente de la plataforma, lo cual es un principio fundamental del middleware.

    \item \textbf{Seguridad y Validación:}
    \begin{itemize}
        \item \textbf{Autenticación:} Al tener variables de sesión, ya estamos ocupando middleware, ya que verificamos la identidad del usuario antes de permitirle tocar los recursos.
        \item \textbf{Validación de Stock:} Esta nueva integración en el proyecto actúa como un \textbf{Middleware de Integración de Recursos}, verificando que los productos mostrados y los que se intentan comprar coincidan con la realidad física de la base de datos, protegiendo la integridad del sistema.
    \end{itemize}
\end{itemize}

    
\subsection{Manejo de procesos e hilos.}

Para abordar el criterio de concurrencia y manejo de procesos, AutoHub implementa un mecanismo de `Web Workers` (hilos de ejecución en segundo plano del navegador) en la capa de presentación, permitiendo simular la ejecución paralela de tareas sin bloquear la interfaz de usuario.

\subsubsection{Arquitectura de concurrencia.}

La implementación utiliza dos hilos (threads) en la capa cliente:

\begin{itemize}
    \item \textbf{Hilo principal (Main Thread):} Controla la interfaz gráfica, procesa eventos del usuario, y actualiza el DOM (estructura de la página).
    \item \textbf{Hilo secundario (Web Worker):} Ejecuta tareas de validación (comprobar stock disponible) sin interrumpir la experiencia del usuario. Este hilo se comunica con el hilo principal únicamente a través de mensajes (postMessage).
\end{itemize}

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
\newpage
\thispagestyle{ARC}

\subsubsection{Flujo de ejecución concurrente.}

Cuando el usuario intenta agregar un producto al carrito, el flujo es:

\begin{enumerate}
    \item El hilo principal solicita el stock a \texttt{getStock.php} mediante \textbf{fetch} (llamada asíncrona).
    \item El servidor devuelve la cantidad disponible de productos.
    \item El hilo principal envía un mensaje al Web Worker: \texttt{stockWorker.postMessage(...)} con la información.
    \item \textbf{Sincronización:} El Web Worker ejecuta la validación en paralelo sin bloquear la UI.
    \item El worker devuelve la respuesta al hilo principal: \texttt{onmessage}.
    \item Si el stock es suficiente, el producto se agrega al carrito; de lo contrario, se muestra una notificación.
\end{enumerate}

Este mecanismo simula una estructura de productor-consumidor (producer-consumer pattern) donde el hilo principal actúa como productor de solicitudes de validación y el worker como consumidor que procesa dichas solicitudes.

\subsubsection{Protección contra condiciones de carrera.}

Aunque la validación inicial ocurre en el cliente (worker), la protección definitiva contra \textbf{race conditions} (conflictos cuando múltiples procesos acceden al mismo recurso) ocurre en el servidor:

\begin{itemize}
    \item En \texttt{Procesar\_pago.php}, antes de actualizar el stock, se realiza una consulta a la base de datos para obtener el valor actual más reciente.
    \item Se compara este valor con la cantidad solicitada; solo si hay suficiente stock, se procede a la actualización.
    \item Este enfoque en capas asegura que, aunque dos usuarios intenten comprar simultáneamente (desde Web Workers diferentes en sus navegadores), la base de datos mantendrá la consistencia del inventario.
\end{itemize}

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
\newpage
\thispagestyle{ARC}

\subsubsection{Componentes involucrados.}

\begin{itemize}
    \item \textbf{worker.js:} Define la lógica del Web Worker. Recibe mensajes del hilo principal y responde con validaciones.
    \item \textbf{script.js:} Comunica eventos del usuario (agregar al carrito) y coordina la sincronización entre el hilo principal y el worker.
    \item \textbf{getStock.php:} Proporciona datos de stock en tiempo real desde la base de datos.
    \item \textbf{Procesar\_pago.php:} Valida nuevamente el stock y realiza la actualización atómica (sin interrupciones) en la base de datos.
\end{itemize}

Esta implementación demuestra los conceptos fundamentales de concurrencia: ejecución paralela de tareas, comunicación entre hilos mediante mensajes, y sincronización para evitar conflictos de acceso a recursos compartidos.

\subsection*{Descripción del funcionamiento:} 
Esta sección implementa la Gestión de Dispositivos de Salida. El script actúa como una interfaz de bajo nivel entre el navegador y el hardware de impresión. Primero, realiza una validación en el Sistema de Archivos mediante la función \textbf{file exists}, asegurando la integridad referencial antes de asignar recursos de memoria. Posteriormente, utiliza un elemento \textbf{iframe} para cargar el recurso PDF en el buffer del navegador. A través de la función \textbf{window.print()}, se envía una señal de interrupción al Sistema Operativo, invocando al Spooler de Impresión y al driver del periféricox para transformar los datos digitales en una salida física.

\subsection*{Descripción del funcionamiento:} 
Este fragmento de código demuestra la administración de la Persistencia de Estado mediante el uso de variables de sesión. El servidor utiliza un espacio de memoria RAM (segmento de datos de la sesión) para almacenar un puntero hacia el último recurso generado en el almacenamiento secundario. El uso de operadores ternarios en PHP permite una gestión dinámica de la interfaz, donde el enlace de impresión se comporta como un objeto condicional: si el puntero existe en la tabla de descriptores de la sesión, se habilita el acceso al archivo; de lo contrario, el recurso se mantiene oculto para prevenir fallos de acceso a archivos no encontrados (File Not Found errors).

\section{Conclusión.}
El desarrollo del proyecto \textbf{AutoHub} ha permitido consolidar los conceptos fundamentales sobre la arquitectura de sistemas operativos, específicamente en las áreas de gestión de memoria, sistemas de archivos y administración de dispositivos de entrada/salida (E/S).

A lo largo del desarrollo, se logró una integración profunda entre los distintos niveles de abstracción del sistema:

\begin{itemize}
\item \textbf{Gestión de Memoria y Estados:} Se diferenció el uso de memoria volátil en el cliente (JavaScript) para la lógica de usuario, frente al uso de segmentos de memoria protegida en el servidor (PHP Sessions). Esta última actúa como un descriptor de recursos que permite la persistencia de estado durante el ciclo de vida de la transacción.
\item \textbf{Administración del Sistema de Archivos:} La implementación de un mecanismo de persistencia basado en archivos PDF en el directorio \texttt{tickets} demuestra el control de nodos en el almacenamiento secundario. La nomenclatura única basada en \textit{timestamps} garantiza la integridad de los datos y evita colisiones en la tabla de archivos del sistema operativo.
\item \textbf{Control de Dispositivos y Comunicación Software-Hardware:} El mayor avance de esta versión es la implementación de un puente de E/S. Mediante llamadas al sistema (\textit{System Calls}) invocadas desde el navegador, el proyecto logra comunicarse con el \textbf{Spooler de Impresión} del sistema operativo host, gestionando la salida física de datos hacia un periférico (Epson) de manera asíncrona.
\item \textbf{Verificacion por medio del Middleware:} La incorporación de \textbf{PHPMailer} y la integración de protocolos de comunicación vía \textbf{JSON} aseguran que el flujo de información no solo sea persistente, sino también transportable entre diferentes procesos del sistema. Asimismo, la configuración de permisos y el manejo de rutas relativas subrayan la importancia de la seguridad y el direccionamiento en entornos de despliegue reales.

En conclusión, \textbf{AutoHub} se consolida como un ecosistema funcional que ejemplifica el ciclo de vida completo del procesamiento de datos: desde la captura en la capa de aplicación, el procesamiento en la capa de servicios, hasta la administración física de hardware. Esta arquitectura no solo optimiza el uso de recursos, sino que cumple con los estándares de interoperabilidad y gestión de periféricos exigidos en la ingeniería de sistemas modernos.

\end{itemize}
